🔑 Authentication

Personal Tokens are tied to your individual user account, expire after 12 hours, and are only suitable for testing on developer.webex.com.

Service Apps use the OAuth 2.0 Client Credentials grant (machine-to-machine). They don't expire on a fixed schedule, support automatic token refresh, and are the correct choice for production integrations. An org admin must approve a Service App before it can access org resources.

Most common causes:

  • Expired token — personal tokens expire after 12 hours. Copy a fresh one.
  • Wrong header format — must be Authorization: Bearer YOUR_TOKEN (capital B, space after Bearer, no quotes around token).
  • Trailing whitespace — copy-pasting often includes invisible newlines. Trim with .trim().
  • Wrong org — your token might belong to a personal org, not the WxCC-enabled org.

For read-only access (dashboards, reporting, monitoring), request:

  • cjp:config_read — read agents, queues, teams, sites
  • cjp:analyzer_read — access historical and real-time reporting
  • spark:people_read — resolve user identities (optional)

Avoid requesting cjp:config (write access) if you only need to read data.

Cache the token with its expiry time and refresh it proactively 60 seconds before expiry. See the token caching example in Getting Started → Service App.

JSON
{
  "postmanEnvironment": {
    "access_token": "{{access_token}}",
    "expires_at": "refresh at least 60 seconds before expiry",
    "refreshPolicy": "request a new token before any API call when expires_at is near"
  }
}

📡 Webhooks

Work through this checklist in order:

  • ✅ Is your server publicly reachable over HTTPS? Use ngrok for local testing.
  • ✅ Does your server respond 200 OK within 5 seconds? WxCC will not retry on timeout.
  • ✅ Is the webhook registration still active? GET /v1/webhooks and check status — should be active.
  • ✅ Are you subscribed to the correct resource and event? Re-check your registration payload.
  • ✅ Is your TLS certificate valid? WxCC won't deliver to endpoints with self-signed or expired certs.

Yes. If your endpoint returns a non-2xx status or times out (>5s), WxCC retries with exponential backoff for up to 24 hours. After 24 hours of failures, the webhook subscription may be automatically disabled. Always respond 200 immediately and process the event asynchronously.

WxCC webhooks are org-level subscriptions — you'll receive events for all agents/queues in your org. To filter, check the relevant field (data.agentId, data.queueId) in your event handler and ignore events that don't match your criteria.

It's an HMAC-SHA1 signature of the raw request body, computed using the secret you provided at registration. Always verify this signature before processing any event — it proves the request genuinely came from Webex and hasn't been tampered with. See Webhooks → Security for the full verification code.

🖥️ Desktop SDK

The SDK communicates with the parent desktop via postMessage — it only works when loaded inside the WxCC agent desktop iFrame. If you open your widget URL directly in a browser tab, init() hangs forever because there's no parent to handshake with.

To test locally: Load the widget through the actual agent desktop (desktop.wxcc-us1.cisco.com). Use the Desktop Layout JSON to point to your ngrok URL during development.

You can add multiple widgets to different areas of the desktop layout — the panel (right side), task area (center), or navigation bar (left). Each widget runs in its own iFrame. There's no hard limit, but consider agent usability — most deployments use 1–3 custom widgets.

Yes — your widget is a web app in an iFrame and can call external APIs through your backend. Keep secrets on the server side, expose an HTTPS endpoint, and configure CORS to allow the widget's origin (*.wxcc-us1.cisco.com).

🔒
Never store secrets in widget frontend code

Frontend JS is visible to anyone. Always proxy sensitive API calls through your backend server.

🔀 Flow Designer

You can configure the HTTP node timeout from 1 to 30 seconds. If the external API doesn't respond within the timeout, the flow takes the "Timeout" output edge. Target under 2 seconds for good caller experience — callers feel every millisecond of IVR delay.

Yes — chain multiple HTTP Request nodes in sequence. For example: node 1 authenticates the caller, node 2 fetches account balance, node 3 logs to an audit system. Keep total external call time under 5–6 seconds combined to avoid making callers wait too long in IVR.

  • Flow Variables — exist only within one flow execution. Use for per-call data (customer name, account ID, intent).
  • Global Variables — shared across all flows in the org. Use for shared config (API tokens, endpoint URLs).
  • CAD Variables — a subset of flow variables attached to the task record and visible on the agent desktop via the SDK.

WxCC Flow Designer maintains a version history of each flow. You can view past versions, compare changes, and roll back from the admin portal. For CI/CD pipelines, use the Flow Management API to publish specific versions programmatically.

📊 Reporting & Analytics

Historical reports and CDRs are available for up to 13 months from the current date. For longer-term storage, export data to your own data warehouse (BigQuery, Snowflake, Redshift) on a regular schedule. See the nightly ETL example.

Real-time stats endpoints return a point-in-time snapshot. For live wallboards, poll every 15–30 seconds. For truly real-time state changes, combine polling with webhooks — webhooks notify you of state changes instantly, polling gives you the latest counts.

❌ Errors & Debugging

Beyond missing scopes, 403 can also mean:

  • Service App not approved — a Webex Org Admin must authorize your Service App. Pending approval returns 403.
  • Feature not enabled — some APIs require specific WxCC features (outbound campaigns, AI) enabled on your subscription.
  • Wrong orgId — you're querying a resource that belongs to a different org than your token's org.
  • Admin-only endpoint — some endpoints require spark-admin role, not just API scopes.
HTTPError Response to Log
HTTP/1.1 403 Forbidden
trackingid: WXCC-abc-123
Content-Type: application/json

{
  "errorCode": "missing_scope",
  "message": "Token does not include the required contact center scope",
  "requiredScope": "cjp:config_read"
}
💡
Always log the trackingId

The trackingid response header is your reference number for Cisco support tickets. Log it for every error.

Two likely causes:

  • Pagination — you're only reading the first page. Use the cursor from the Link header to fetch all pages.
  • Site/team scoping — some endpoints filter by site or team by default. Try adding or removing siteId param to get org-wide results.

⭐ Best Practices
  • 🔒 Never hardcode tokens — use environment variables or a secrets manager (AWS Secrets Manager, HashiCorp Vault).
  • 🔒 Verify webhook signatures — always validate X-Spark-Signature on every incoming request.
  • 🔒 Use Service Apps, not personal tokens — personal tokens are tied to individuals and expire.
  • 🔒 Principle of least privilege — request only the scopes you actually need.
  • 🔒 No secrets in widget code — frontend JS is visible to anyone. Proxy sensitive calls through your backend.
  • 🔒 Rotate secrets regularly — rotate webhook secrets and API credentials on a schedule.
  • ⚡ Cache aggressively — agent lists, queue configs, and org data rarely change. Cache for 30–60 seconds.
  • 🔁 Implement retry with backoff — all API calls should retry on 429 and 5xx errors.
  • ⏱️ Set request timeouts — never make an API call without a timeout (recommend 10s max).
  • 📊 Monitor your webhook endpoint — track delivery success rate; alert if it drops below 99%.
  • 🧪 Test against a sandbox org — use a non-production WxCC org to validate changes before deploying.
  • 📝 Log trackingIds — always log the trackingid header for debugging with Cisco support.

Combine webhooks (server-side) with the Desktop SDK (client-side) for the best result:

  • Webhook listener on your server catches task:created events and fetches CRM data immediately — before the agent even picks up.
  • Desktop SDK widget on the agent desktop subscribes to AgentContactOffered and calls your backend to retrieve the pre-fetched CRM data.
  • This two-pronged approach means CRM data is ready before the agent answers — no visible loading state.

See Desktop SDK → Screen Pop and Webhooks → Listener for the implementation details.
