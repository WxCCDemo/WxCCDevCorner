🔑 Authentication

Q1 What's the difference between a Personal Token and a Service App?
+

Personal Tokens are tied to your individual user account, expire after 12 hours, and are only suitable for testing and development on developer.webex.com.

Service Apps use the OAuth 2.0 Client Credentials grant (machine-to-machine). They don't expire on a fixed schedule, support automatic token refresh, and are the correct choice for production integrations. An org admin must approve a Service App before it can access org resources.

Q2 My token works on developer.webex.com but not in my code — why?
+

The most common causes:

  • Expired token — personal tokens expire after 12 hours. Copy a fresh one.
  • Wrong header format — must be Authorization: Bearer YOUR_TOKEN (capital B, space after Bearer, no quotes around the token).
  • Trailing whitespace — copy-pasting tokens often includes an invisible newline. Trim it with .trim().
  • Wrong org — your token might belong to a personal org, not the WxCC-enabled org.
Q3 Which scopes do I need for a read-only integration?
+

For read-only access (dashboards, reporting, monitoring), request these scopes:

  • cjp:config_read — read agents, queues, teams, sites
  • cjp:analyzer_read — access historical and real-time reporting
  • spark:people_read — resolve user identities (optional)

Avoid requesting cjp:config (write access) if you only need to read data.

Q4 How do I handle token refresh automatically?
+

Cache the token with its expiry time and refresh it proactively 60 seconds before expiry. See the token caching example in the Getting Started — Service App section.

⬡ Token refresh pattern
// Refresh 60 seconds before expiry
const isExpired = Date.now() >= tokenCache.expiresAt - 60_000;
if (isExpired) await refreshToken();

🔌 REST APIs

Q5 How do I find my Org ID?
+

Three ways to get your Org ID:

  • API call: GET https://webexapis.com/v1/organizations — the id field in the first item is your Org ID.
  • WxCC Admin portal: Log in to admin.webex.com → Organization Settings — the Org ID is shown at the top.
  • Developer portal: At developer.webex.com, click your avatar → the "My Profile" page shows your org.
Q6 Can I create agents in bulk via API?
+

There is no native bulk-create endpoint. You'll need to loop over a list and call POST /contactCenter/agents for each agent individually. Add a small delay (100–200ms) between each call to avoid rate limiting. The Bulk Create Teams example in REST APIs → Configuration shows this pattern.

⚠️

The agent's Webex user account must already exist before you can create a WxCC agent profile. Use the Webex Provisioning API or admin.webex.com to create the user first.

Q7 How do pagination cursors work?
+

WxCC list endpoints return a Link response header with a rel="next" URL when more pages exist. To fetch all results, keep requesting the next URL until it's absent. Never use offset-based pagination — it's not supported. See the auto-paginate helper in REST APIs → Pagination.

📡 Webhooks

Q8 My webhook endpoint isn't receiving events — how do I debug?
+

Work through this checklist in order:

  • ✅ Is your server publicly reachable over HTTPS? Use ngrok for local testing.
  • ✅ Does your server respond 200 OK within 5 seconds? WxCC will not retry if it times out.
  • ✅ Is the webhook registration still active? GET /v1/webhooks and check status field — should be active.
  • ✅ Are you subscribed to the correct resource and event? Re-check your registration payload.
  • ✅ Is your TLS certificate valid? WxCC won't deliver to endpoints with self-signed or expired certs.
Q9 Does WxCC retry failed webhook deliveries?
+

Yes. If your endpoint returns a non-2xx status or times out (>5s), WxCC retries with exponential backoff for up to 24 hours. After 24 hours of failures, the webhook subscription may be automatically disabled. Always respond 200 immediately and process the event asynchronously.

Q10 Can I receive webhook events for a specific agent or queue only?
+

WxCC webhooks are org-level subscriptions — you'll receive events for all agents/queues in your org. To filter by agent or queue, check the relevant field (data.agentId, data.queueId) in your event handler and ignore events that don't match your criteria.

Q11 What is the X-Spark-Signature header and why does it matter?
+

It's an HMAC-SHA1 signature of the raw request body, computed using the secret you provided when registering the webhook. Always verify this signature before processing any event — it proves the request genuinely came from Webex and hasn't been tampered with. See the full verification code in Webhooks → Security.

🖥️ Desktop SDK

Q12 My widget loads but the SDK init() never resolves — what's wrong?
+

The SDK communicates with the parent desktop via postMessage — it only works when loaded inside the WxCC agent desktop iFrame. If you open your widget URL directly in a browser tab, init() will hang forever because there's no parent to handshake with.

To test locally: Load the widget through the actual agent desktop (desktop.wxcc-us1.cisco.com). Use the Desktop Layout JSON to point to your ngrok URL during development.

Q13 How many custom widget panels can I add to the desktop?
+

You can add multiple widgets to different areas of the desktop layout — the panel (right side), the task area (center), or the navigation bar (left). Each widget runs in its own iFrame. There's no hard limit on count, but consider agent usability — too many panels create cognitive overload. Most deployments use 1–3 custom widgets.

Q14 Can my widget make API calls to external systems?
+

Yes — your widget is just a web app in an iFrame. It can call any external API (your CRM, knowledge base, AI service, etc.) using standard fetch or axios calls. Make sure your backend endpoints have CORS configured to allow the widget's origin (*.wxcc-us1.cisco.com).

⚠️

Never store API keys or secrets in the widget's frontend code. Always proxy sensitive calls through your backend server.

Q15 How do I pass data from the flow (call variables) into my widget?
+

Flow variables set as Call-Associated Data (CAD) are automatically passed to the agent desktop when the task is assigned. Your widget can read them via the SDK's task event data:

⬡ Read flow variables in widget
Desktop.agentContact.addEventListener('eAgentContact', (event) => {
  if (event.detail.type === 'AgentContactOffered') {
    const { callVariables } = event.detail.data;
    // callVariables contains everything set in your Flow Designer
    console.log(callVariables.customerTier);   // "Enterprise"
    console.log(callVariables.accountId);      // "ACC-98765"
  }
});

🔀 Flow Designer

Q16 What's the timeout limit for the HTTP Request node?
+

You can configure the HTTP node timeout from 1 to 30 seconds. If the external API doesn't respond within the timeout, the flow takes the "Timeout" output edge. Keep your external APIs fast — callers on hold experience any delay directly. Target under 2 seconds for good caller experience.

Q17 Can I call multiple external APIs in a single flow?
+

Yes — you can chain multiple HTTP Request nodes in a flow. For example: node 1 authenticates the caller, node 2 fetches their account balance, node 3 logs the interaction to an audit system. Each node parses its own response into flow variables that downstream nodes can use.

Keep the total external call time under 5–6 seconds combined to avoid making callers wait too long in IVR.

Q18 What's the difference between a Flow Variable and a Global Variable?
+
  • Flow Variables — exist only within a single flow execution. Created in Flow Designer per-flow. Use these for data that's specific to one call (customer name, account ID, intent).
  • Global Variables — shared across all flows in the org. Configured in WxCC Admin. Use these for shared config values like API endpoint URLs or authentication tokens that don't change per-call.
  • CAD Variables (Call-Associated Data) — a special subset of flow variables that get attached to the task record and are visible on the agent desktop.
Q19 Can I version-control my flows?
+

WxCC Flow Designer maintains a version history of each flow. You can view past versions, compare changes, and roll back to a previous version from the admin portal. For CI/CD pipelines, use the Flow Management API to publish specific versions programmatically.

📊 Reporting & Analytics

Q20 How far back can I query historical data?
+

Historical reports and CDRs are available for up to 13 months from the current date. For longer-term storage, export data to your own data warehouse (BigQuery, Snowflake, Redshift) on a regular schedule. See the nightly ETL example in Reporting → Scheduled Export.

Q21 How often is real-time data updated?
+

Real-time statistics endpoints return a point-in-time snapshot — the data reflects the state at the moment of your API call. WxCC doesn't push real-time updates via the stats API. For live wallboards, poll every 15–30 seconds. For truly real-time state changes, combine polling with webhooks — webhooks notify you of state changes instantly.

Q22 Can I get call recordings via API?
+

Yes — if call recording is enabled on your WxCC instance, you can retrieve recording metadata and download URLs via GET /v1/contactCenter/recordings. The actual audio files are stored in Cisco's secure storage and accessed via time-limited signed URLs returned by the API. Requires the cjp:analyzer_read scope and the recording feature to be enabled in your org.

❌ Errors & Debugging

Q23 I get a 403 even though I have the right scope — what else could it be?
+

Beyond missing scopes, 403 can also mean:

  • Service App not approved — a Webex Org Admin must authorize your Service App. Pending approval returns 403.
  • Feature not enabled — some APIs require specific WxCC features (e.g., outbound campaigns, AI) to be enabled on your org's subscription.
  • Wrong orgId — you're querying a resource that belongs to a different org than your token's org.
  • Resource-level restriction — some endpoints are admin-only and require the spark-admin role, not just API scopes.
Q24 How do I enable detailed error logging for my integration?
+

Add an Axios interceptor to log all requests and responses:

⬡ Axios error interceptor
axios.interceptors.response.use(
  res => res,
  err => {
    const { status, data, headers } = err.response || {};
    console.error({
      status,
      errorCode:  data?.errorCode,
      message:    data?.message,
      trackingId: headers?.['trackingid'],  // include in support tickets!
      url:        err.config?.url
    });
    return Promise.reject(err);
  }
);
// Always include the trackingId when contacting Cisco support
Q25 The API returns data but it's missing some agents/queues — why?
+

Two likely causes:

  • Pagination — you're only reading the first page. The default max is often 100. If you have more resources, use the cursor from the Link header to fetch all pages.
  • Site/team scoping — some endpoints filter by site or team by default. Check if your token's user is scoped to specific sites and try adding siteId as a parameter or removing it to get org-wide results.

⭐ Best Practices

Q26 What are the top security best practices for WxCC integrations?
+
  • 🔒 Never hardcode tokens — use environment variables or a secrets manager (AWS Secrets Manager, HashiCorp Vault).
  • 🔒 Verify webhook signatures — always validate X-Spark-Signature on every incoming request.
  • 🔒 Use Service Apps, not personal tokens — personal tokens are tied to individuals and expire.
  • 🔒 Principle of least privilege — request only the scopes you actually need.
  • 🔒 No secrets in widget code — frontend JS is visible to anyone. Proxy sensitive calls through your backend.
  • 🔒 Rotate secrets regularly — rotate webhook secrets and API credentials on a schedule.
Q27 How should I structure my integration for production reliability?
+
  • ⚡ Cache aggressively — agent lists, queue configs, and org data rarely change. Cache for 30–60 seconds.
  • 🔁 Implement retry with backoff — all API calls should retry on 429 and 5xx errors.
  • ⏱️ Set request timeouts — never make an API call without a timeout (recommend 10s max).
  • 📊 Monitor your webhook endpoint — track delivery success rate; alert if it drops below 99%.
  • 🧪 Test against a sandbox org — use a non-production WxCC org to validate changes before deploying.
  • 📝 Log trackingIds — always log the trackingid header for debugging with Cisco support.
Q28 What's the recommended architecture for a CRM screen-pop integration?
+

The recommended pattern combines webhooks (server-side) with the Desktop SDK (client-side):

  • Webhook listener on your server catches task:created events and fetches CRM data immediately (before the agent even picks up)
  • Desktop SDK widget on the agent desktop subscribes to AgentContactOffered and calls your backend to retrieve the pre-fetched CRM data
  • This two-pronged approach means data is ready before the agent answers — no waiting

For the implementation, see Desktop SDK → Screen Pop and Webhooks → Listener.
