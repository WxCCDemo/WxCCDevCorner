🔀 Flow Designer

Flow Designer & Intelligent Routing

Build sophisticated IVR flows, call routing logic, and customer journeys using WxCC Flow Designer — with HTTP integrations to external systems baked in.

What is Flow Designer?

Flow Designer is the visual drag-and-drop builder inside WxCC Admin that defines what happens when a contact arrives — from initial greeting, through IVR menus, external data lookups, to final agent routing. Every flow is made of nodes connected by edges.

💡

Extensibility in Flows

  • HTTP Request node — call any external REST API mid-flow (CRM lookup, authentication, AI scoring)
  • Flow variables — pass data between nodes and into agent desktop call variables
  • Custom widget trigger — pop widgets on the agent desktop based on flow data
  • Subflows — reusable flow segments that can be imported across multiple main flows

Flow Types

📞

Inbound Voice Flow

Handles incoming calls. Plays greetings, collects IVR input, routes to agents.

📤

Outbound Campaign Flow

Drives outbound dialing campaigns with predefined scripts and routing.

💬

Digital Channel Flow

Handles chat, email, and social contacts through routing and bot escalation.

🔁

Subflow

Reusable logic blocks (e.g., authentication, CRM lookup) shared across flows.

🌐 HTTP Request Node

The HTTP Request node lets your flow call any external REST API. Use it to authenticate callers, fetch customer data, score intent with AI, or log to an external system.

🌐 HTTP Node — Sample CRM Authentication Request

Here's what the external API server (your Node.js backend) might look like that the HTTP node calls:

⬡ Your Backend API (called by HTTP Node)
// Express route that WxCC Flow HTTP Node calls
app.post('/api/verify-account', async (req, res) => {
  const { accountNumber } = req.body;

  // Look up in your database
  const customer = await db.customers.findOne({ accountNumber });

  if (!customer) {
    return res.status(404).json({ authenticated: false });
  }

  // Return structured data for Flow variable parsing
  res.json({
    authenticated:   true,
    customerName:    customer.name,
    tier:            customer.tier,          // 'Enterprise' | 'Standard'
    openCases:       customer.openCases,
    accountId:       customer.id,
    preferredAgent:  customer.csm?.agentId   // preferred CSM agent if any
  });
  // WxCC will parse $.customerName → FlowVar_Name, etc.
});

📦 Flow Variables

Variables store data across nodes in a flow and can be passed to the agent desktop as call variables.

Variable TypeScopeExample Use
Flow VariablesCurrent flow onlyStore HTTP response data mid-flow
Global VariablesAll flows in orgAPI tokens, shared config values
Call Variables (CAD)Passed to agent desktopShow customer data in desktop screen pop
Predefined VariablesSystem-setcallerId, channelType, entryPointId
💡

Reference syntax

In HTTP node bodies and headers, reference variables as $(VariableName). In Condition node expressions, use the variable name directly: FlowVar_Tier == "Enterprise".

🔱 Condition Node & Branching

The Condition node evaluates a flow variable and routes to different branches. Chain conditions to build complex logic.

⚙️ Flow Management API

Flows can also be managed programmatically via the REST API — useful for CI/CD pipelines that auto-deploy flow updates.

⬡ List all flows
const response = await axios.get('https://webexapis.com/v1/contactCenter/flows', {
  headers: { Authorization: `Bearer ${token}` },
  params: { orgId: ORG_ID }
});
response.data.items.forEach(f => {
  console.log(`${f.name} [v${f.version}] — Status: ${f.publishStatus}`);
});
⬡ Publish (activate) a flow version
// Publish a specific flow version to make it live
await axios.post(
  `https://webexapis.com/v1/contactCenter/flows/${flowId}/publish`,
  { version: 3, comment: 'Added enterprise priority routing' },
  { headers: { Authorization: `Bearer ${token}`, 'Content-Type': 'application/json' } }
);
console.log('Flow published and now live');
