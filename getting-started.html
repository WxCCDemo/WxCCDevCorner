Authentication & Setup

Everything you need to authenticate with the Webex Contact Center API — OAuth 2.0 flows, Service Apps, scopes, and your first API call.

WxCC APIs use OAuth 2.0 Bearer tokens. This guide covers all three token methods — personal tokens for testing, OAuth for user-facing apps, and Service Apps for production server-to-server integrations.

👤

Prerequisites

👤

Webex Account

A Webex org with Contact Center enabled. Admin or developer role required.

🌐

Developer Portal

Access to developer.webex.com — free registration, no credit card needed.

⬡

Postman or API Client

Any HTTP-capable tool works. Examples use Postman-style requests with headers and JSON bodies.

🔒

HTTPS Endpoint

Required only for webhooks. Use ngrok for local development.

🔑

Authentication Methods

⚠️
Development Only

Personal tokens expire after 12 hours. Never use them in production systems.

  1. Go to developer.webex.com

    Sign in with your Webex account credentials.

  2. Click your avatar → Copy Personal Access Token

    The token is shown on your profile page under "My Webex Apps".

  3. Use in Authorization header

    Add Authorization: Bearer <token> to every request.

Use when building apps where end-users grant your app permission to act on their behalf (e.g., a supervisor dashboard).

  1. Create an Integration

    At developer.webex.com → My Webex Apps → Create an Integration. Set your redirect URI.

  2. Redirect user to authorize URL

    Direct users to the Webex authorization endpoint with your client_id and desired scopes.

  3. Exchange code for token

    After user approves, exchange the returned code for an access + refresh token pair.

✅
Recommended for Production

Service Apps use machine-to-machine tokens with configurable expiry and automatic refresh. No user interaction required.

  1. Create a Service App

    developer.webex.com → My Webex Apps → Create a Service App. Choose scopes carefully.

  2. Organization Admin Authorizes

    A Webex Org Admin must approve the Service App. This grants access to org-level resources.

  3. Use Client Credentials Grant

    POST to the token endpoint with your client_id and client_secret to get a long-lived access token.

🔄

OAuth 2.0 Authorization Code Flow

sequenceDiagram participant App as Your App participant Auth as Webex Auth Server participant User as User / Browser participant API as WxCC REST API App->>User: 1. Redirect to /authorize?client_id=...&scopes=... User->>Auth: 2. Show login + consent screen Auth->>User: 3. User approves User->>App: 4. Redirect with ?code=... App->>Auth: 5. POST /token {code, client_id, client_secret} Auth->>App: 6. access_token + refresh_token App->>API: 7. Call WxCC APIs with Bearer access_token API->>App: 8. JSON response data

Use the refresh_token to get a new access_token before expiry — avoiding re-authorization

⚙️

Service App Token (Production)

For server-to-server integrations, use the Client Credentials flow — no user interaction required. Cache the token to avoid fetching on every request.

HTTP Service App Token Request 
POST https://webexapis.com/v1/access_token

Headers
Authorization: Basic {{base64_client_id_client_secret}}
Content-Type: application/x-www-form-urlencoded

Body
grant_type=client_credentials
scope=cjp:config cjp:config_read spark:people_read
💡
Token Refresh Strategy

The cache checks expiresAt - 60_000 — refreshing 60 seconds early prevents any request from using an expired token due to clock skew or network latency.

🎯

Your First API Call

Validate your token and org access with a safe read endpoint before building further.

HTTP 
GET https://webexapis.com/v1/contactCenter/agents?orgId={{orgId}}&max=5

Headers
Authorization: Bearer {{access_token}}
Accept: application/json
application/json 
{
  "items": [
    {
      "id": "agent-001",
      "name": "Ava Chen",
      "agentState": "Available",
      "teamName": "Care Team"
    }
  ]
}
POSTMAN VARIABLES 
{
  "orgId": "your-webex-org-id",
  "access_token": "paste-service-app-or-personal-token",
  "base64_client_id_client_secret": "base64(clientId:clientSecret)"
}
❌

Common Errors & Solutions

Cause: The Bearer token is missing, malformed, or has expired.

  • Check that the Authorization header is spelled correctly (capital A)
  • Personal tokens expire after 12 hours — refresh or use a Service App
  • Ensure no extra whitespace around the token value — use .trim()

Cause: Your token doesn't have the required scope for the operation.

  • Check the API docs for required scopes on the specific endpoint
  • Re-authorize with the additional scope
  • For Service Apps, ensure the admin has approved the new scope
  • Service App not yet approved by org admin also returns 403

Cause: You've exceeded the API rate limit.

  • Check the response headers for the recommended retry-after wait time
  • Cache stable configuration data (agent lists, queue configs) to reduce repeated calls
  • Implement exponential backoff for retries

Cause: The resource ID is wrong, or you're querying the wrong org.

  • Verify the Org ID in your request params
  • Ensure IDs are URL-encoded if they contain special characters
  • Confirm the resource exists with a list call first
