Auth Foundation — Secure API Access

flowchart LR A[Developer / App] -- OAuth --> B[Webex Identity] B -- Token --> C[WxCC APIs] C -- JSON response --> A style B fill:#00476B,color:#fff style C fill:#6366f1,color:#fff

Business benefit: Gives teams confidence that integrations authenticate safely and retrieve contact center data with the right scopes.

What this sample teaches

  • Create an integration or service-app access pattern at developer.webex.com
  • Request scopes for config, people lookup, and reporting use cases
  • Call task, agent, queue, or site endpoints with a Bearer token
  • Implement token refresh / caching for longer-running production integrations

Run / adapt this sample

  1. Configure environment

    Set WXCC_TOKEN (or client credentials) and WXCC_ORG_ID as environment variables.

  2. Call a safe read endpoint

    Validate token, org access, and scopes with the agents or queues endpoint first.

  3. Move to Service App

    Replace personal token with the service-app token cache pattern for production use.

JavaScriptFirst WxCC API Call
const token = process.env.WXCC_TOKEN;
const orgId = process.env.WXCC_ORG_ID;

const res = await fetch(
  `https://webexapis.com/v1/contactCenter/agents?orgId=${orgId}&max=5`,
  { headers: { Authorization: `Bearer ${token}` } }
);

const data = await res.json();
console.log(`✅ Token valid — ${data.items.length} agents returned`);
data.items.forEach(a => console.log(` • ${a.name} (${a.agentState})`));
✅
Expected Result

Token accepted → 200 OK → Agent list returned with name and state. Now you have a working foundation for every other integration.

Real-time Action — Webhook Listener

flowchart LR A[WxCC Event] -- HTTPS POST --> B[Webhook Listener] B -- Verify HMAC --> C{Valid?} C -- Yes --> D[CRM / Alert / Email] C -- No --> E[Reject 401] style D fill:#10b981,color:#fff style E fill:#f43f5e,color:#fff

Business benefit: Lets teams respond to customer moments immediately — email alerts, CRM updates, compliance logs, recording workflows.

Implementation checklist

  • Create webhook subscription for the exact event types your workflow needs.
  • Expose an HTTPS endpoint, using ngrok only for local development.
  • Verify the signature before trusting the payload.
  • Return 200 OK quickly, then process the event asynchronously.
  • Store event IDs to avoid duplicate processing during retries.
JavaScriptExpress Webhook Listener
app.post('/webhooks/wxcc', express.raw({ type: 'application/json' }), (req, res) => {
  // 1. Verify HMAC-SHA1 signature
  const sig  = req.headers['x-spark-signature'];
  const hmac = require('crypto').createHmac('sha1', process.env.WEBHOOK_SECRET);
  const expected = hmac.update(req.body).digest('hex');
  if (sig !== expected) return res.status(401).send('Invalid signature');

  // 2. Respond 200 immediately — never delay this
  res.sendStatus(200);

  // 3. Process asynchronously
  const event = JSON.parse(req.body.toString());
  queueEventForProcessing(event);
});
JSONEvent Processing Shape
{
  "eventId": "evt_12345",
  "resource": "task",
  "eventType": "task:ended",
  "data": {
    "taskId": "task-abc",
    "queueId": "sales-queue",
    "agentId": "agent-001",
    "ani": "+14155550100",
    "wrapupCode": "Resolved"
  }
}

Expected output

  • Listener logs signature verified, event queued, and crm updated.
  • Supervisor receives alert only for selected event types.
  • Duplicate event IDs are ignored rather than creating repeated CRM notes.

Business benefit: Improves agent productivity — customer context visible before the agent even answers, zero tab switching.

flowchart LR A[Incoming Contact] --> B[Agent Desktop] B --> C[Custom Widget iFrame] C --> D[Desktop JS SDK Events] C --> E[CRM Lookup] E --> F[Customer Context Panel] style C fill:#00476B,color:#fff style F fill:#10b981,color:#fff

Implementation checklist

  • Host the widget on HTTPS and register the widget URL in desktop layout JSON.
  • Initialize the Desktop JS SDK and subscribe to contact lifecycle events.
  • Use ANI, DNIS, or call variables to look up CRM context.
  • Write disposition notes or outcome fields back after contact end.
JavaScriptScreen Pop on Incoming Call
await Desktop.config.init({ widgetName: 'CRM Widget' });

Desktop.agentContact.addEventListener('eAgentContact', async ({ detail }) => {
  if (detail.type === 'AgentContactOffered') {
    const customer = await crm.lookup(detail.data.callerId);
    if (customer) renderCustomerContext(customer);
  }
  if (detail.type === 'AgentContactEnded') {
    clearContext();
  }
});

{

            "title": "CRM Context"
          }
        }]
      }
    }
  }
}

Expected output

  • When a call is offered, the widget shows customer name, tier, open cases, and suggested next action.
  • When the contact ends, the widget clears context and optionally posts wrap-up data to CRM.

Implementation checklist

Intelligent Routing — Flow HTTP Lookup

Business benefit: Smarter routing and shorter resolution paths — Gold customers skip the queue, standard callers get AI-predicted routing.

flowchart LR A[Flow HTTP Request Node] --> B[Lookup API] B --> C[CRM / DB] C --> B B --> D[JSON Response] D --> E[Flow Variables] E --> F{Tier / Intent} F --> G[Priority Queue] F --> H[Standard Queue] style B fill:#00476B,color:#fff style G fill:#10b981,color:#fff

Implementation checklist

  • Design the HTTP response around flow variables, not backend object names.
  • Always return a safe fallback queue and error reason.
  • Keep lookup latency low; Flow Designer should not wait on slow CRM operations.
  • Log correlation IDs so failed lookups can be traced from flow to backend.
JavaScriptFlow Lookup Backend API
app.post('/api/route-caller', async (req, res) => {
  const account = await db.account(req.body.accountNumber);
  res.json({
    tier:  account.tier,
    queue: account.tier === 'Gold' ? 'priority-queue-id' : 'standard-queue-id',
    name:  account.name,
    openCases: account.openCases
  });
  // WxCC Flow Designer parses $.tier → FlowVar_Tier, $.queue → FlowVar_Queue
});
JSONFlow Response Mapping
{
  "tier": "Gold",
  "queue": "priority-queue-id",
  "name": "Ava Chen",
  "openCases": 2,
  "fallbackQueue": "standard-queue-id",
  "lookupStatus": "matched"
}

Expected output

  • Flow variable FlowVar_Tier becomes Gold.
  • Condition node routes Gold callers to priority queue and unmatched callers to fallback queue.

Config Automation — Configuration Management

Business benefit: Consistent configuration at scale — partner onboarding, admin automation, governance audits.

flowchart LR A[Admin / Partner Tool] --> B[Config API Client] B --> C[Queues] B --> D[Teams] B --> E[Entry Points] B --> F[CSV / JSON Export] F --> G[Review / Migration / Audit] style B fill:#00476B,color:#fff style G fill:#10b981,color:#fff

Implementation checklist

  • Export current configuration before making changes.
  • Use IDs for API calls and names only for human review tables.
  • Validate required fields before bulk-create or bulk-update operations.
  • Produce a dry-run report showing what will be created, updated, or skipped.
JavaScriptConfig Export to CSV
const { data } = await wxcc.get('/contactCenter/queues', { orgId: ORG_ID });
const rows = data.items.map(q => ({
  name:         q.name,
  channel:      q.channelType,
  serviceLevel: q.serviceLevelThreshold
}));
await writeCsv('queues.csv', rows);
console.log(`✅ Exported ${rows.length} queues to queues.csv`);
JavaScriptDry-run Change Plan
const desired = await readCsv('queues-desired.csv');
const current = await wxcc.getQueues({ orgId: ORG_ID });

const plan = desired.map(row => {
  const existing = current.find(q => q.name === row.name);
  return {
    action: existing ? 'update' : 'create',
    name: row.name,
    targetServiceLevel: row.serviceLevel
  };
});

console.table(plan);

Expected output

  • CSV export lists queue name, channel, service level, and routing metadata.
  • Dry-run plan clearly separates create, update, and skip actions before any write call.

Desktop APIs — Agent State & Contact Monitor

Business benefit: Gives operations teams a controlled way to observe agent/contact state and build supervisor tooling outside the standard desktop.

flowchart LR A[Service App] --> B[Desktop REST API] A --> C[WebSocket Notifications] C --> D[Agent State Stream] C --> E[Contact Event Stream] D --> F[Supervisor Tool] E --> F style C fill:#00476B,color:#fff style F fill:#10b981,color:#fff

Implementation checklist

  • Use REST calls for point-in-time state and WebSocket events for live changes.
  • Track agent ID, session ID, contact ID, and event timestamp together.
  • Do not use monitoring tools to bypass supervisor permissions or business policy.
JavaScriptSubscribe to Desktop Events
const socket = new WebSocket(process.env.WXCC_DESKTOP_EVENTS_URL, {
  headers: { Authorization: `Bearer ${process.env.WXCC_TOKEN}` }
});

socket.onmessage = message => {
  const event = JSON.parse(message.data);
  if (event.type === 'agent.state.changed') {
    stateStore.upsertAgent(event.agentId, {
      state: event.state,
      reason: event.reasonCode,
      changedAt: event.timestamp
    });
  }
};

Expected output

  • Supervisor tool updates agent state changes without manual refresh.
  • Contact events can be correlated with queue, agent, and wrap-up outcomes.

Callback Automation — Web Form to WxCC Callback

Business benefit: Reduces queue frustration by letting customers request a callback from a website, CRM case, or mobile app.

flowchart LR A[Customer Form] --> B[Callback API Service] B --> C[Validate Phone / Consent] C --> D[WxCC Callback Request] D --> E[Queue / Team] E --> F[Agent Calls Back] style B fill:#00476B,color:#fff style F fill:#10b981,color:#fff

Implementation checklist

  • Validate phone number, timezone, and consent before creating the request.
  • Map web form intent to the right queue, team, or entry point.
  • Write callback status back to the CRM or customer case.
JavaScriptCreate Callback Request
app.post('/api/callbacks', async (req, res) => {
  const request = {
    phoneNumber: req.body.phoneNumber,
    customerName: req.body.name,
    queueId: mapIntentToQueue(req.body.intent),
    preferredTime: req.body.preferredTime,
    attributes: {
      crmCaseId: req.body.caseId,
      source: 'web-form'
    }
  };

  const callback = await wxcc.createCallback(request);
  res.status(201).json({ callbackId: callback.id, status: 'created' });
});

Expected output

  • Customer receives confirmation with callback reference.
  • WxCC creates a callback routed to the correct queue.
  • CRM case stores callback ID for follow-up tracking.

Recording Workflow — QA and Compliance Archive

Business benefit: Moves recording metadata and availability events into QA, compliance, and audit workflows without manual supervisor effort.

flowchart LR A[Recording Available Event] --> B[Webhook Listener] B --> C[Recording Metadata API] C --> D[Secure Archive] C --> E[QA Review Queue] D --> F[Audit / Compliance] style B fill:#00476B,color:#fff style F fill:#10b981,color:#fff

Implementation checklist

  • Trigger from recording-related events instead of polling aggressively.
  • Store metadata, retention policy, and review status separately from media files.
  • Limit archive access by role and log every retrieval.
JavaScriptQueue Recording for Review
async function processRecordingEvent(event) {
  const recording = await wxcc.getRecording(event.data.recordingId);

  await archive.storeMetadata({
    recordingId: recording.id,
    contactId: recording.contactId,
    agentId: recording.agentId,
    queueName: recording.queueName,
    startedAt: recording.startedAt
  });

  await qaQueue.add({
    recordingId: recording.id,
    reason: 'new-recording',
    priority: recording.queueName === 'complaints' ? 'high' : 'normal'
  });
}

Expected output

  • New recordings appear in QA review with contact, agent, queue, and timestamp metadata.
  • Compliance archive can prove when recording metadata was received and who accessed it.
