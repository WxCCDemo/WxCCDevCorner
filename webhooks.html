Webhooks & Real-Time Events

React to contact center events the moment they happen — no polling required. Register a URL and WxCC pushes HMAC-verified event payloads to you in real time.

On This Page
  • ⚡ How Webhooks Work
  • 🔄 Event Flow Diagram
  • 📋 Event Types
  • 📝 Register a Webhook
  • 🖥️ Build a Listener
  • 🔒 HMAC Security
  • ⚙️ Manage Webhooks

Instead of polling, webhooks let WxCC push events to your server the moment something happens. You register an HTTPS endpoint, and WxCC sends a POST with a JSON payload — verified with HMAC-SHA1 — whenever a subscribed event fires.

⚡

How Webhooks Work

⚡

Real-Time

Events delivered within milliseconds of occurring — no polling delay.

🎯

Selective

Subscribe only to the event types you care about — filter by resource and action.

🔒

Secure

HMAC-SHA256 signature on every payload — verify it's genuinely from Webex.

🔁

Retryable

WxCC retries failed deliveries with exponential backoff for up to 24 hours.

🔄

Event Delivery Flow

sequenceDiagram participant C as 📞 Customer participant W as WxCC Platform participant A as Agent Desktop participant S as Your Server participant CRM as CRM / Database C->>W: Calls contact center W->>W: Route to available agent W->>A: Assign task to agent W-->>S: POST /webhook {event: "task:created", taskId, callerId} S->>CRM: Look up caller by phone number CRM-->>S: Return customer record S->>A: Push screen-pop via Desktop SDK A->>W: Agent accepts call W-->>S: POST /webhook {event: "task:assigned", agentId, taskId} S->>CRM: Log interaction start A->>W: Agent wraps up W-->>S: POST /webhook {event: "task:ended", duration, wrapUpCode} S->>CRM: Update case with call outcome

Register a Webhook

💡
Local Development Tip

Use ngrok http 3000 to expose your local server publicly. WxCC will deliver events to the ngrok URL during testing.

HTTPRegister Webhook — application/json
POST https://webexapis.com/v1/webhooks

Headers
Content-Type: application/json
Authorization: Bearer {{access_token}}

Body
{
  "name": "WxCC Integration - Task Events",
  "targetUrl": "https://your-server.com/webhooks/wxcc",
  "resource": "task",
  "event": "all",
  "orgId": "{{orgId}}",
  "secret": "{{webhook_secret}}"
}

Building a Webhook Listener

✅
Always respond 200 first

Return HTTP 200 immediately before any async processing. If WxCC doesn't get a 200 within 5 seconds, it marks the delivery as failed and retries.

HTTPIncoming Webhook Delivery — application/json
POST https://your-server.com/webhooks/wxcc

Headers
Content-Type: application/json
X-Spark-Signature: {{hmac_signature}}

Body
{
  "id": "evt_12345",
  "name": "task:ended",
  "resource": "task",
  "orgId": "{{orgId}}",
  "data": {
    "taskId": "task-abc",
    "callerId": "+14155550100",
    "agentId": "agent-001",
    "wrapUpCode": "RESOLVED"
  }
}

Expected listener response
HTTP/1.1 200 OK
Content-Type: application/json

{ "status": "received" }
🔒

HMAC Security & Verification

flowchart LR A["WxCC sends POST\n+ X-Spark-Signature header"] --> B["Your server receives\nraw request body"] B --> C["Compute HMAC-SHA1\nof raw body using secret"] C --> D{"Computed HMAC\n== header value?"} D -- Yes --> E["✅ Process event\nReturn 200 OK"] D -- No --> F["❌ Reject — 401\nLog potential spoofing"] style E fill:#10b981,color:#fff style F fill:#f43f5e,color:#fff
🔒
Never skip verification in production

Without HMAC verification, any actor who discovers your webhook URL can send fake events. Always validate every incoming request before processing.
